我的第一個Google Extensions套件 - 頁面之間的溝通

Posted on 2022-02-17 | In Google extension

實際運作邏輯

套件主要的溝通方式主要會由三個js組成 background、conetent、popup，其餘js數量沒有限制

  1. background : 長時間的待命，主要負責extension邏輯與功能，只要套件有開啟的狀態就是持續待命，簡單說不論你在哪一個分頁逛哪一個頁面，它都癡癡的等待你喚醒執行交代的工作
  2. content : 專職跟頁面溝通的腳色，沒有它就無法取得頁面中的資料
  3. popup : 使用者介面的呈現，可透過 extension API 讀取、操作 content.js

第二步：制作幕後主要溝通的頭頭 background.js


我們在 background.js 輸入 console.log 確認是否有成功載入

在這次設計套件中，主要我希望使用者在點擊套件後，會執行一個新的分頁，並且將之後需要的資料都集中在這個分頁中來操作，所以我們需要點擊以及開啟後的頁面內容，內容我們會寫在popup.html內。

第三步：使用者頁面 popup.html

我們先把使用者頁面內容製作起來

刷新一下我們的套件看看有沒有成功



第四步：抓目標頁面資訊 content_scripts

先回頭看先前設定的 manifest.json

  1. matches : 符合的網址，可以替換任何想執行的網址
  2. js : 執行的js檔案，js名稱可以自行取
  3. run_at : 默認”document_idle”，試情況可更換”document_start” or “document_end”

接下來，我們的目標是要抓到 Google 首頁的 Logo， 顯示在我們的使用者頁面頁面內


  1. 在 content_scripts 中可以透過 chrome.runtime.sendMessage 發送資料，希望在網頁完整讀取後就傳送

  2. 在 popup.js 中可以透過 chrome.runtime.onMessage.addListener 接收資料，當 content_scripts 所加載的分頁完畢後就會傳入

我的第一個Google Extensions套件 - manifest安裝檔

Posted on 2022-02-16 | In Google extension

跨出的第一步

公司是使用shopline電商平台，每當連續假期過完後準備出貨，總是會聽到出貨小幫手們的吶喊，
因為…寫不完的郵寄地址阿…除了手痠還是手痠，更是浪費時間(1張寫2分鐘 100張訂單真的會寫死人)。

而我們其實也有熱感印機的存在，不過都僅用於超商的條碼列印，畢竟人家都直接產生好了麻，不像郵寄訂單只有地址。雖然有幫忙簡單用Execl抓一下內容可以處理掉手痠的問題，但數量多的時候…Ctrl+C > Ctrl+V 小拇指也很酸…


那有什麼方法是每台電腦都可以做到，每個人都可以簡單使用，又可以實時更新功能並不用一個一個跑過去手動安裝的呢??


所以在學習JavaScript一段時間後，Google Extensions根本是完全符合我的選項，而第一個套件在工作之餘摸索了好一段時間，人生第一個公開軟件好不容易正式上線了，在還沒忘記之前來好好記錄一下心路歷程


第一步：制作擴充功能的安裝檔 manifest.json

每個套件都會擁一個 manifest，主要提供套件內容、版本、說明、權限等重要資訊，我們可以簡單說就是告訴Google這個extensions要做什麼。

官方提供起手式

照著修改一下

  • name - 套件名稱
  • description - 套件的說明
  • version - 在擴充店上架時才會用到的，每次一次更新都需要變化數字來確認版本
  • manifest_version - 現在最新版本為V3
  • background - 躲在幕後，只做重要事情的頭頭
  • content_scripts - 頭頭的小弟，也是跟頁面的主要聯絡窗口
  • permissions - 要啟用的權限

資料參考

content_scripts : https://developer.chrome.com/docs/extensions/mv3/content_scripts/#static-declarative
